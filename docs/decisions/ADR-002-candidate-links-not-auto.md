# ADR-002: 核心链路不自动断定——候选+证据+用户确认

**状态**: 已接受
**日期**: 2026-07-12
**决策人**: AI Agent（基于架构设计约束）
**关联**: 红线 R4、架构设计文档

## 背景

iTestAgent 的核心差异化能力之一是「先理解项目再测试」。自然期望是 Agent 自动分析代码、自动识别核心用户链路（如「启动→登录→搜索→下单」），然后自动测试。

然而代码中不存在「运行时用户旅程」这条边。从静态代码能拿到的只有 View 名称、Router 声明、文案资源、埋点名等弱信号。LLM 基于这些信号拼出的链路「看着对但常错」，且会给出高置信度的错误结论——这比不推断更危险。

## 决策

**Project Analyzer 不自动断定核心链路。只输出「候选链路 + 证据来源 + 置信度」，用户在 TUI 中勾选、排序、修正后才生效。**

## 实施

1. Project Profile 中 `features` 字段的每个条目必须包含 `evidence`（证据来源，如 `README#登录`、`LoginViewController.swift`、`埋点 login_click`）和 `confidence`（置信度）
2. TUI 中展示候选链路列表，用户可勾选确认、修改、排序
3. 只有用户确认的链路进入 TestPlan 和 Flow
4. 未确认的推断链路不得当作既定事实（红线 R4）

## 备选方案

| 方案 | 优点 | 缺点 | 结论 |
|---|---|---|---|
| **全自动断定** | 用户体验好，一键测试 | 准确率不可控，假阳性损害信任 | ❌ 不选 |
| **候选+用户确认** | 可控、可审计、信任不崩塌 | 用户需额外操作 | ✅ 选择 |
| **不做链路推断** | 零风险 | 失去核心差异化价值 | ❌ 不选 |

## 后果

### 正面
- 用户始终在控制链路选择，不会有「Agent 测了不该测的东西」
- 候选链路带证据，用户可验证推断逻辑
- 随着历史 Flow 积累，未来可增强推断准确率

### 负面
- 用户需要手动确认，不是「一句话全自动」
- confidence 评分缺乏标准校准集，初期可能不准
