# iTestAgent 开发计划安排文档

## 0. 文档说明

本计划基于全部文档（用户故事与验收标准、架构设计、技术选型、数据流全链路、AI Native 开发手册、开发避坑手册）编制，给出**分阶段、可执行、可验收**的开发排期与任务拆解。

配套文档：

```
iTestAgent 全量用户故事与验收标准规格书
iTestAgent 架构设计文档
iTestAgent 技术选型文档
iTestAgent 数据流全链路技术说明文档
iTestAgent AI Native 开发理念与实战技巧手册
iTestAgent 开发避坑与关键注意点手册
```

排期单位约定：

人力：1 名独立开发者（全栈，AI Native 全程） 相对周 W 按单人全职折算，兼职按投入比例线性放大 里程碑以可验收结果为准，不以代码量为准

## 1. 总体策略

核心原则（来自避坑手册与可行性分析）：

```
1. 先 Spike 后功能：先验证真机链路与元素可定位性，再堆能力
2. 去风险 MVP：人在环路记录器 + 稳健性能趋势工具
3. 研究级能力后置：自动链路推断/自动断言/FPS 精确/xctrace 深解析 标 Deferred/实验性
4. 每阶段以 P0 验收为门槛，P1 允许已知限制并声明
5. AI Native 全流程：规格即上下文、小步可验证、真机必实测
```

阶段总览：

```
Phase 0  立项与多 Backend 横评（决定路线）
Phase 1  骨架与环境（TUI/Server/doctor/devices）
Phase 2  项目分析与 TestPlan（Project Profile + 计划确认）
Phase 3  真机执行核心（构建安装 + XCUITest + DeviceBackend 探索录制）
Phase 4  证据/性能/报告（稳健性能子集 + 三件套报告 + 归因）
Phase 5  打磨与 MVP 验收（Flow 重放/重跑/安全/门禁）
Phase 6+ 增强（xctrace 深解析/断言增强/链路推断/团队与云扩展）
```

## 2. Phase 0：立项与多 Backend 横评（约 W1-W2）

目标：用最小成本验证各层 backend 候选，选出 MVP 最优组合。

核心原则：**不预设单一工具路线。** 每一层都做 2-4 候选横评，用 Phase 0 spike 数据决定 MVP 主力 backend。

任务：

```
T0.1 建仓与工程规范
  - monorepo(Bun)、AGENTS.md、schemas/、fixtures/、mock backends 脚手架

T0.2 Device backend 横评（最高优先）
  - Spike A: mobile-mcp 路线 — 设备发现、安装/启动 app、UI tree、截图、tap/swipe/type
  - Spike B: Appium/WDA 路线 — WDA 签名安装、Appium session、page source、screenshot、gesture
  - Spike C: iphone-use 路线 — iPhone Mirroring + WebRTC + 视觉 fallback（仅当 A/B 失败时）
  - Spike D: mock backend — fixtures 模拟所有 backend 输出，保障无真机开发和 CI

T0.3 Performance backend 横评
  - Spike P1: XcodeTraceMCP / @xctrace-analyzer/core — trace export、hitches/hangs、not_exportable
  - Spike P2: instrumentsmcp — 录制+分析+report 一体化
  - Spike P3: raw xcrun xctrace export — 最小自研 fallback

T0.4 TUI backend 横评
  - OpenTUI: 验证长日志、Markdown、工具调用卡片、Bun 打包
  - ~~Rezi~~: 已排除（npm rezi 是 2015 CSS 工具，非 TUI 框架）
  - Ink: 验证最小交互版是否足够支撑 MVP

T0.5 Project analyzer 横评
  - XcodeQuery CLI: 验证 JSON 输出是否足够稳定（T0.5 结果：本机不可用，降级为 optional future）
  - XcodeProj + xcodebuild: 传统方案作为对照

T0.6 技术验证结论
  - 输出决策矩阵 + Phase 0 横评报告
  - 确定 MVP 各层 backend 选择
```

里程碑 M0：

```
- device backend 至少一路在真机上跑通（截图+UI tree+launch time）
- performance backend 至少一路能获取 hitches/hangs
- TUI backend 至少一路能跑通交互式 Shell
- 决策矩阵明确：每层选 MVP 主力 + fallback
出口标准：M0 未过则暂停功能开发，先解决真机/性能瓶颈
```

决策矩阵：

| 如果 | 则 |
|---|---|
| mobile-mcp 真机稳定且企业环境可用 | MVP 优先 mobile-mcp device backend |
| mobile-mcp 被 EDR 或 mobilecli 阻断 | AppiumBackend 做主线，mobile-mcp 保留可选 |
| WDA UI tree 质量差 | 研究 iphone-use 视觉 fallback |
| XcodeTraceMCP core 稳定 | 性能模块优先复用 core |
| XcodeTraceMCP core 不适合依赖 | raw-xcrun-minimal fallback |
| OpenTUI 不稳定 | Ink fallback（T0.4 已验证 16/16 通过） |
| XcodeQuery CLI JSON 稳定 | ~~优先作为 Project Profile backend~~ → T0.5 不可用，raw xcodebuild + Tuist/XcodeProj 优先 |

里程碑 M3：

```

- 断言分层，未定不判 passed
```

## 12. 单人独立开发者工作方式

单人 + AI Native 分工：人管决策，AI 管产出。人负责真机 spike 实测、签名与 backend setup 与设备处理、架构决策、AC 验收、危险操作确认、文档定稿；AI 负责读规格、出计划、写实现与测试、生成文档草稿。策略：串行推进；mock+fixtures 无真机也能写逻辑；真机任务集中验证；能复用绝不自研；研究级能力后置；每个 US 小步交付。

## 14. 风险登记与应对

| 风险                | 阶段  | 影响       | 应对                                       |
| ------------------- | ----- | ---------- | ------------------------------------------ |
| 真机签名/backend 跑不通 | P0/P3 | 阻塞全局   | M0 先验证；doctor 引导；backend 预配置 |
| 无测试项目定位差  | P0/P3 | 探索失效   | 定位 spike；退纯录制；对话流式断言         |
| xctrace 跨版本变动  | P4    | 性能中断   | 薄封装+容错；主推 hitches；保留原始 .trace |
| AI 过度自信         | P2/P3 | 错误结论   | 候选+证据+确认；诚实降级                   |
| 可复现 vs 探索矛盾  | P3    | 结果不可信 | 固化 Flow 才复现；探索显式标注             |
| 范围蔓延            | 全程  | 延期       | 研究级后置；P0 门槛；backend 横评定路线       |
| 子进程泄漏          | P3/P4 | 稳定性     | server 统一生命周期与回收                  |
