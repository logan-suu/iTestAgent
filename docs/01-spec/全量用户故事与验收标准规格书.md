# iTestAgent 全量用户故事与验收标准规格书

## 文档目的

本规格书把产品能力拆解为**可执行、可验收的用户故事（User Story）与验收标准（Acceptance Criteria）**，用于指导开发、测试和验收。

- 面向对象：产品、研发、QA
- 产品定位：Local-first、TUI-first、真机与 iOS Simulator 同级支持的测试 Agent
- 目标用户：iOS 客户端开发者本地自测与失败复现
- 参考验证项目示例：无既有测试、需自动构建签名的 iOS 项目

## 用语与约定

```
US-x.y     用户故事编号
AC         验收标准（Acceptance Criteria）
Given/When/Then  场景化验收写法
MVP        第一版必做
Deferred   推迟到后续版本
Experimental 实验性能力，标注后可交付但不作为硬门槛
```

优先级定义：

```
P0  MVP 硬门槛，缺失则第一版不可交付
P1  MVP 应做，允许有已知限制
P2  增强项，可推迟
```

## E1 安装与环境诊断（doctor）

### US-1.1 安装 CLI（P0）

作为 iOS 开发者，我希望能快速安装 iTestAgent CLI，以便在本机使用。

验收标准：

```
AC1 支持通过 npm 或 Homebrew 安装 itestagent
AC2 安装后 itestagent --version 能输出版本号
AC3 安装不要求登录任何账号
```

### US-1.2 环境诊断（P0）

作为开发者，我希望运行 itestagent doctor 就能知道本机能否跑真机测试。

验收标准：

```
Given 本机已安装 itestagent
When 运行 itestagent doctor
Then 检查并逐项报告：Xcode / Command Line Tools / xcodebuild / xcrun devicectl / xcrun xctrace / xcresulttool / Appium / Appium XCUITest Driver / WebDriverAgent 可构建可签名 / 证书与 provisioning profile / iPhone 连接 / Developer Mode / 设备信任 / WDA 启动与端口转发
AC1 每一项给出 通过 / 失败 / 需手动处理 三态
AC2 失败项给出可执行的修复指引（命令或手机端操作步骤）
AC3 不因单项失败而中断整体诊断，输出完整清单
AC4 诊断结果可被 engine 读取用于阻止不满足前置条件的执行
```

### US-1.3 首次真机跑通引导（P0）

作为首次使用者，我希望 doctor 能引导我完成签名、Developer Mode、设备信任、backend 前置条件配置。

验收标准：

```
AC1 doctor 能识别“签名不可用 / Developer Mode 未开 / 设备未信任 / backend 未就绪”并分别引导
AC2 引导为分步骤、可复制命令 + 手机端操作说明
AC3 明确告知首次跑通可能需要用户配合 15-30 分钟
AC4 引导完成后可复检并显示已通过
```

## E2 设备发现与真机准备（含 Simulator）

### US-2.1 列出本机设备（P0）

作为开发者，我希望运行 itestagent devices 看到本机连接的 iPhone。

验收标准：

```
Given 至少一台 iPhone 通过 USB 连接
When 运行 itestagent devices
Then 列出每台设备的 名称 / 型号 / iOS 版本 / UDID / 电量 / 状态
AC1 状态至少区分 healthy / untrusted / busy / developer_mode_off
AC2 无设备时给出明确提示与连接指引
AC3 数据来源使用 xcrun devicectl list devices
```

### US-2.2 设备健康检查（P1）

作为开发者，我希望执行前 Agent 能确认设备可用。

验收标准：

```
AC1 执行前自动 healthcheck：连接、信任、Developer Mode（physical）、backend 可用、runtime/simctl（simulator）
AC2 不满足时不进入执行，给出原因和修复建议（区分 physical/simulator 原因）
AC3 healthcheck 结果记录到 run metadata（含 targetKind）
```

### US-2.3 选择执行目标 — 真机与 Simulator（P0，ADR-011）

**作为** iOS 开发者，**我希望**在 CLI/TUI 中同时看到本机 iPhone 真机和可用的 iOS Simulator，**以便**根据当前需求选择执行目标。

验收标准：

```
AC1 itestagent devices 同时列出真机（devicectl）和 Simulator（simctl list --json）
AC2 每个设备显示 KIND（physical/simulator）、NAME、OS/RUNTIME、UDID、STATE
AC3 TestPlan 中 device 字段必须包含 targetKind（physical|simulator）
AC4 跨 targetKind 切换需要用户确认（不能静默 fallback）
AC5 不支持的目标类型标记为 blocked，给出原因
```

## E3 项目分析与 Project Profile

### US-3.1 全量项目扫描（P0）

作为开发者，我希望 Agent 启动后先全量扫描并理解我的项目。

验收标准：

```
Given 在 iOS 项目目录中启动 itestagent
When Agent 完成项目分析
Then 扫描 源码结构（View/ViewController/SwiftUI View/ViewModel/Router/Coordinator/Deep Link/feature modules）、文案资源、埋点名、网络 API 名、README/docs/注释、已有测试与 scheme/test plan
AC1 工程结构识别使用 XcodeProj + xcodebuild -list/-showBuildSettings，结果确定可靠
AC2 Swift 结构使用 swift-syntax；符号/语义使用 sourcekit-lsp/SourceKitten
AC3 默认不读取 .gitignore 命中项、secrets、DerivedData 等敏感/无关文件
AC4 分析结果生成 Project Profile
```

### US-3.2 生成 Project Profile（P0）

作为开发者，我希望分析结果沉淀为结构化 Project Profile。

验收标准：

```
AC1 Project Profile 至少包含 app(name/bundleId/workspace/scheme)、features、testAssets、suggestedSmoke
AC2 默认存储 ~/.itestagent/projects/<project-hash>/project-profile.json
AC3 仅在用户确认“固化到项目”后写入 <project>/.itestagent/project-profile.json
AC4 Profile 可被 TestPlan 引用
```

### US-3.3 带证据的候选核心链路（P1）

作为开发者，我希望 Agent 给出候选核心链路，但由我确认，而不是自动断定。

验收标准：

```
AC1 Agent 输出“候选核心链路 + 置信度 + 证据来源”（如 README 第X节 / LoginViewController / 文案“下单” / 埋点 checkout_click）
AC2 在 TUI 中用户可勾选、排序、修正候选链路
AC3 只有用户确认的链路进入 TestPlan 和 Flow
AC4 系统不把未确认的推断链路当作既定事实
```

### US-5.1 编译 TestPlan（P0）

作为开发者，我希望自然语言最终变成结构化 TestPlan。

验收标准：

```
AC1 自然语言、TUI 操作、CLI 辅助命令都编译为统一 TestPlan
AC2 TestPlan 至少包含 target/device(targetKind+selector)/appSource/execution/features/testData/assertion/flows/metrics/performance(baselineDomain)/artifacts/report
AC3 TestPlan 可审计、可复现、可重跑
AC4 TestPlan 引用 Project Profile
```

### US-5.2 展示计划并确认（P0）

作为开发者，我希望执行前看到并确认计划。

验收标准：

```
Given Agent 生成了 TestPlan
When 展示给用户
Then 显示设备、执行方式、用例/链路、采集项、基线、预计耗时
AC1 提供 开始 / 修改 / 取消

## E6 App 构建/安装/启动

### US-6.1 确定 App 来源（P0）

作为开发者，我不想手动准备包，希望 Agent 自动从当前 workspace 得到可安装 App。

验收标准：

```
AC1 默认 App 来源为当前 workspace（如某 iOS 项目的 .xcworkspace）
AC2 来源优先级：用户显式指定 app/ipa > 当前 workspace 已有构建产物 > 自动 xcodebuild 构建
AC3 无既有产物时自动构建（识别 scheme/target/bundleId/签名）
AC4 无法确定时在 TUI 询问用户
AC5 用户体验上不需要先理解“App 安装来源”概念
```

### US-6.2 构建与安装到真机（P0）

作为开发者，我希望 Agent 构建并安装 App 到本机 iPhone。

验收标准：

```
AC1 使用 xcodebuild 构建，devicectl 安装
AC2 构建日志经 xcbeautify 可读化
AC3 签名失败时给出明确原因和修复建议，不静默失败
AC4 安装成功后可 launch/terminate/背景切换/deep link
```

### US-6.3 Simulator 构建安装（P0，ADR-011）

作为开发者，我希望 Agent 能将 App 构建为 Simulator SDK .app 并安装到指定 iOS Simulator。

验收标准：

```
AC1 使用 xcodebuild 构建 Simulator SDK .app（platform=iOS Simulator,id=<UDID>）
AC2 通过 simctl install/launch/terminate 管理 Simulator App 生命周期
AC3 BuildDriver 根据 targetKind 自动选择 destination（physical: platform=iOS,id=UDID / simulator: platform=iOS Simulator,id=UDID）
AC4 错误明确区分：构建失败 vs Simulator runtime 缺失 vs App slice 不匹配
AC5 不能将真机 .ipa 安装到 Simulator 或 Simulator .app 安装到真机
```

验收标准：

```

## E8 测试执行：DeviceBackend 探索路径

### US-8.1 无测试代码时探索执行（P0）

作为无测试的开发者，我希望 Agent 能直接在真机或 Simulator 上探索执行。

验收标准：

```
Given 项目无 XCUITest
When 选择 DeviceBackend 探索
Then engine -> DeviceBackend -> UI actions -> run steps -> flow -> report
AC1 通过可插拔 DeviceBackend 接口调用底层工具（如 Appium、mobile-mcp），不自研
AC2 能执行 点击/输入/滑动/系统弹窗处理/截图/读取 UI tree
AC3 探索过程完整记录为 run steps
AC4 元素定位不稳定时给出降级说明，不假装成功
```

### US-8.2 交互式录制（P1）

作为开发者，我希望 Agent 建议下一步、我确认，逐步固化流程（人在环路）。

验收标准：

```
AC1 Agent 每步给出建议动作，用户可确认/纠偏
AC2 确认后的步骤固化为可重放 Flow
AC3 探索式执行默认不保证可复现，只有固化成 Flow 后才可复现
```

## E12 性能采集与 baseline
    US-12.1~12.3

### US-12.1 采集核心性能指标（P0）

作为开发者，我希望采集关键性能指标。

验收标准：

```
AC1 覆盖 launch time / memory peak(近似) / crash / test duration / hitches/hangs
AC2 FPS 作为近似指标（FPS-like），不承诺精确实时 FPS
AC3 深度 xctrace summary 解析为实验性，第一版保留原始 .trace 供人工打开
AC4 底层用 xcrun xctrace export（--toc 探测 + --xpath 抽取），对 schema 做跨 Xcode 版本容错
AC5 memory peak 标注为近似值
```

### US-12.2 本地 baseline 与趋势（P1）

作为开发者，我不想配阈值，希望自动建立 baseline 并看趋势。

验收标准：

```
AC1 不要求用户配置性能阈值
AC2 首次成功 run 自动建立本地 baseline；功能失败或 crash 的 run 不建立 baseline
AC3 后续 run 与 baseline 对比输出变化趋势
AC4 用户可在 TUI 接受某次结果为新 baseline（高风险操作需确认）
AC5 明确失败仅限 crash / 功能失败 / 执行失败
```

## E17 Agent 编排循环与权限引擎

### US-17.1 编排循环（P0）

作为系统，我需要一个稳定的 Agent 执行循环。

验收标准：

```
AC1 循环：stream LLM -> 执行 tool-call -> 等结算 -> 有工具调用则续跑 -> 无调用且无 pending 输入则 idle
AC2 使用 Vercel AI SDK 多步 tool-calling 实现
AC3 iPhone 真机能力封装为 MCP tools
AC4 同一设备/session 串行，不同设备可并发
AC5 不引入 Effect / 事件溯源等重型依赖
```

### US-17.2 权限引擎（P0）

作为开发者，我希望危险操作被拦截并需确认。

验收标准：

```
AC1 规则模型 { action, resource, effect: allow|deny|ask } + 通配符匹配
AC2 高风险操作默认 ask：清 App 数据 / 卸载或重装 App / 写入项目目录 / 保存账号或 token / 接受 run 为新 baseline / 生成或覆盖 Flow / 生成测试代码草稿
AC3 ask 时阻塞等待用户确认；用户可选 always 记住规则并持久化
AC4 deny 直接阻止；用户拒绝时停止本次循环
```

## 去风险与非目标

去风险优先级（先做地基）：

```
先做：doctor 引导 + 结构化 Project Profile + 交互式录制 Flow + 稳健性能子集(launch/memory/crash/hitches) + 报告
先验证：端到端真机 spike（构建+签名+backend setup+session+UItree+launch time）
先验证：Simulator spike（simctl lifecycle+Simulator SDK build+Appium/WDA session+UI tree/actions，G5-SIM）
先验证：示例项目元素定位 spike（对话主界面 accessibility 树质量）
```

明确非目标（Deferred / 不作为第一版硬门槛）：

```
自动判定核心用户链路（改为候选+用户确认）
全自动探索 + 自动判 passed（改为人在环路）
FPS 精确指标（只做 FPS-like 近似）
xctrace summary 稳定解析（保留原始 .trace，v2 深化）
测试代码 draft 自动入库（实验性、手动确认）
中心化服务 / 云真机 / 团队共享 / 权限系统
```

### 第一版明确不做

```
不做中心化服务
不做远程 Mac worker pool
不做多用户设备租约
不做云真机矩阵
不做复杂 Web Dashboard
不做团队级 baseline 共享
不做权限系统
不做完整 Test Management 集成
不做 Slack/钉钉 Bot 自动触发
不自动提交测试代码
不把生成的测试草稿自动视为正式测试用例
不在未确认时写入项目目录
不要求用户手动配置性能阈值
```
