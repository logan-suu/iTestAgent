# iTestAgent 技术选型文档

## 1. 目的与范围

本文件给出 iTestAgent 各层技术选型的**决策、候选对比、选择理由、风险与替代方案**，用于研发落地和评审。

- 产品形态：Local-first、TUI-first、只面向 iPhone 真机的测试 Agent
- 目标用户：iOS 客户端开发者本地自测与失败复现
- 参考验证项目示例：无既有测试、需自动构建签名的 iOS 项目
- 选型基准日期：2026-07；跨 Xcode 版本兼容以 Xcode 26 为参照

## 2. 选型原则

```
1. 复用优先：底层能力尽量复用成熟 repo，自研聚焦编排与语义层
2. 官方稳定优先：优先 Apple 官方 CLI（xcodebuild/devicectl/xctrace/xcresulttool）
3. 主流轻量优先：Agent/LLM/工具协议用主流标准（AI SDK + MCP），不自研底层
4. 本地优先：单机可用，不依赖服务器/云/登录
5. 可维护优先：不碰私有框架、私有二进制格式、停更工具
6. 诚实降级：研究级能力（探索/断言/FPS/xctrace summary）以尽力而为 + 明确降级
```

选型标记：

```
[采用]   直接依赖/集成
[借鉴]   参考其实现，不整包依赖
[自研]   无成熟替代，必须自建
[不用]   明确排除，附原因
```

## 4. 语言与运行时

决策：TypeScript + Bun [采用]

候选对比：

| 方案              | 优点                                                         | 缺点                                 | 结论                                       |
| ----------------- | ------------------------------------------------------------ | ------------------------------------ | ------------------------------------------ |
| TypeScript + Bun  | 启动快、原生 TS、内置 bundler/test/包管理、易分发单文件 binary、与 OpenCode 生态一致 | 部分原生模块兼容性需验证             | 采用                                       |
| TypeScript + Node | 生态最广、企业兼容好                                         | 启动慢、需额外 transpile/loader      | 备选/兼容回退                              |
| Swift             | 与 iOS 工具链最近                                            | Agent/TUI/LLM/MCP 生态薄，跨平台弱   | 仅用于必要的原生子进程                     |
| Python            | 脚本方便、部分解析库多                                       | CLI Agent/TUI/流式交互体验弱、分发重 | 仅个别 adapter（如 pymobiledevice3）子进程 |

理由：iTestAgent 是 CLI/TUI Agent，重交互、重工具调用、需易分发，Bun/TS 最契合，且能直接复用 AI SDK + MCP + OpenTUI 生态。

风险与缓解：Bun 长时间子进程管理（xcodebuild/appium/xctrace）需认真做信号转发与回收；必要时保留 Node 兼容运行路径。

## 5. 交互层：CLI 与 TUI

决策：采用可插拔 TuiShell 架构。CLI 用 Commander（轻量入口）；TUI 多候选横评，不默认锁死单一框架。

| 优先级 | 工具 | 判断 | 结论 |
|---|---|---|---|
| 1 | OpenTUI + Solid | TS/Bun 取向、原生渲染、流式输出、键盘交互、Markdown/代码块 | **MVP 第一候选**。验证长日志、工具卡片、Bun 打包 |
| 2 | Rezi | TS/Bun TUI 框架，组件/routing/forms/tables/logs/测试工具完整，仍 beta | **强挑战者**。Phase 0 做同屏布局和日志流 spike |
| 3 | Ink（React TUI） | 成熟度最高，React 心智，AI CLI 采用信号强 | **稳定 fallback**。若 OpenTUI/Rezi 不稳定，Ink 能降低框架风险 |
| 4 | TermUI / Glyph | AI components、ToolCall、StreamingText 产品化能力强 | 借鉴 UI 组件设计 |
| — | Commander | 简单成熟 CLI parser | `itestagent doctor/devices/config` 默认 |
| — | Clipanion | 类型化命令类 | 命令复杂后考虑 |
| — | Oclif | 插件型 CLI 平台强但偏重 | 只有开放 CLI 插件生态时引入 |

理由：产品是 TUI-first，但不应绑定单一框架。iTestAgent 定义自己的 TuiShell 接口和 UI view model，OpenTUI/Rezi/Ink 都只是 renderer。

## 6. 编排与 LLM

决策：自建 Agent 编排循环 [自研] + Vercel AI SDK [采用] + MCP TS SDK [采用]。架构上保留 AgentRuntime 接口，不绑定单一 agent framework。

候选对比（编排实现方式）：

| 优先级 | 工具 | 判断 | 结论 |
|---|---|---|---|
| 1 | Vercel AI SDK | TS agent loop、streaming、tool calling、provider-neutral，最适合 MVP | **第一候选**。验证 tool approval、abort、multi-step、TUI stream |
| 2 | MCP TypeScript SDK | 官方协议栈，stdio/http、client/server | **MCP 底座**。用于工具注册和 adapter 协议 |
| 3 | Mastra | agent/workflow/memory/observability/MCP/Studio 一体化 | Phase 0 对照。若 workflow/observability 收益大，用于增强路线 |
| 4 | LangGraph JS | durable graph、interrupt、HITL、resume | TestPlan/RunPlan 状态复杂化后备选。MVP 不强推 |
| 5 | OpenAI Agents SDK | OpenAI-first，sessions/guardrails/MCP/approval | 适合 OpenAI-only 版本，不适合 provider-neutral 主路线 |
| 6 | Effect | cancellation、retry、resource scope、typed errors 强 | runner/supervisor 子系统局部引入，不全局 Effect 化 |
| 7 | XState | 显式状态机强 | Run lifecycle、Permission 状态建模，不替代 LLM loop |

> 关键结论：OpenCode 编排内核 private/未发布/绑 Effect+SQLite 事件溯源，无法 import。iTestAgent 用 AI SDK 多步 tool-calling 自建循环，约 1/10 复杂度获 90% 收益。

## 7. 工具协议：MCP

决策：MCP TypeScript SDK [采用]

```
- iPhone 真机操作（devicectl/WDA/Appium/截图/日志/xctrace）封装为 MCP tools
- 工具抽象 { description, inputSchema, outputSchema, execute } + 输出大小限制
- 支持本地 stdio server；engine 用统一 tool 接口消费
- backend 以 MCP 形式插拔，engine 不直接拼命令
```

理由：MCP 是工具生态标准，AI SDK 原生消费，解耦 engine 与底层工具，便于后续接入 Appium MCP/XcodeBuildMCP 等参考实现的工具。

## 8. 存储与配置

决策：bun:sqlite + Drizzle [采用]（metadata）+ 文件系统（artifacts）+ JSONC [采用] + macOS security CLI [采用]（secrets）。engine 只依赖 repository interface，ORM 类型不泄漏到上层。

| 优先级 | 工具 | 判断 | 结论 |
|---|---|---|---|
| 1 | bun:sqlite | Bun 内置，无 native npm 安装风险 | **SQLite driver 第一候选**。验证 WAL、busy_timeout、打包后路径 |
| 2 | Drizzle | 类型安全 schema + migrations，Bun/SQLite 适配好 | **ORM 第一候选** |
| 3 | Kysely | SQL builder 清晰，运行时更轻 | Drizzle 主要挑战者。若 migration/schema 需更显式，考虑 Kysely |
| 4 | Zod 4 | TS inference、JSON Schema 转换、生态成熟 | **schema/validation 第一候选**。所有对外 JSON/YAML contract |
| 5 | TypeBox / Valibot | TypeBox JSON Schema-first，Valibot 更轻 | Zod 横评对照 |
| 6 | macOS `security` CLI | Keychain 一等公民，无 native npm 风险 | **SecretStore 第一候选**。封装 add/find/delete generic password |
| 7 | SQLite FTS5 | 日志/报告/历史 run 搜索 | artifact 索引增强 |
| — | keytar | 已归档且 native module 风险高 | 不建议默认使用 |
| — | Prisma | 生态强但偏重，本地 CLI/TUI 不划算 | 不建议 MVP 使用 |
| — | 事件溯源(Event Sourcing) | OpenCode 用法 | 不用（本地单机过度设计） |

存储布局：

```
~/.itestagent/{config,db,projects,sessions,flows,baselines,runs}
runs/<run_id>/{plan.yaml,summary.md,result.json,artifact-index.json,artifacts/}
```

## 9. 真机执行技术栈

决策：采用可插拔 DeviceBackend 架构，不绑定单一工具。底层经由 DeviceBackend 接口调用，实现可替换。

### 9.1 DeviceBackend 候选排序

| 优先级 | Backend | 判断 | Phase 0 建议 |
|---|---|---|---|
| 1 | `mobile-mcp` | 跨平台 iOS/Android MCP，真机/模拟器，UI tree/截图/tap/swipe/type/录屏一站式 | **MVP 第一候选**。Phase 0 作为对照基线跑通 |
| 2 | `@blitzdev/iphone-mcp` | 产品形态最接近"AI 使用 iPhone"，自动 setup/scan UI/action/live viewer | 对照工具。评估首次连接、WDA 自动安装、UI tree 质量 |
| 3 | official `appium-mcp` / raw Appium | WebDriver 标准协议，WDA/Appium 正统路径，日志体系成熟 | 长期标准路线。与 mobile-mcp 横评后决定主路径 |
| 4 | `wda-mcp` | WDA 直连能力强，适合远程/Tailscale 控制方案 | 借鉴远程控制和端口转发经验 |
| 5 | `iphone-use` | iPhone Mirroring + WebRTC + 输入注入，WDA 不可用时的视觉 fallback | Phase 6+ 研究。WDA tree 差时作为备选 |
| 6 | `mobile-device-mcp` | 多设备并行、统一 UI tree、tap/scroll/type 架构有吸引力 | 借鉴 device abstraction 设计 |
| 7 | MockBackend | fixtures 模拟所有 backend 输出 | **MVP 必须有**。保障无真机开发和 CI |

### 9.2 底层工具选型

| 工具 | 定位 | 纳入方式 |
|---|---|---|
| WebDriverAgent | 真机 UI 自动化底层核心桥 | 所有真机 backend 的公共底座 |
| pymobiledevice3 | 最强底层工具箱（syslog/crash/tunnel） | doctor/device/performance adapter 通过子进程调用 |
| go-ios | 生产级设备管理（install/launch/tunnel） | pymobiledevice3 的 fallback |
| XcodeBuildMCP | build/test/log 工具 schema 参考 | 借鉴 xcodebuild/devicectl 封装方式 |
| libimobiledevice/iproxy | classic usbmux/lockdown 工具链 | WDA 端口转发等场景 |

### 9.3 排除

```
[不用] Maestro 作为本地真机核心（本地真机支持弱，仅借鉴 flow 语义）
[不用] Detox 作为通用底座（偏 RN 模拟器）
[不用] ios-simulator-mcp（只面向 Simulator，不是真机）
```

真机前置约束（选型必须配套 doctor 引导）：

```
签名/Provisioning、Developer Mode、设备信任、WDA 首次安装
个人证书 7 天过期、bundle id 冲突、WDA 单独签名等坑
```

## 10. 项目分析技术栈

决策：分层 analyzer，结构识别用官方/成熟解析，语义可选增强，链路只做候选。

| 优先级 | 工具 | 判断 | 结论 |
|---|---|---|---|
| 1 | `xcodebuild -list/-showBuildSettings` | Apple 官方事实源 | **必用**。不要只读 pbxproj 推断 |
| 2 | XcodeProj / Tuist XcodeProj | `.xcodeproj` 解析成熟 | **Project graph 第一候选** |
| 3 | XcodeQuery | GraphQL 风格查询，直接输出 JSON | **强烈建议 spike**。更适合 TS/Bun |
| 4 | SwiftSyntax | Swift 结构解析最佳 | **Source layer 第一候选** |
| 5 | tree-sitter-objc | ObjC legacy fallback | 有 ObjC 项目时加入 |
| 6 | SourceKit-LSP / SourceKitten | 语义增强、符号、定义、引用 | 可选。需已有 build/index，不阻塞 MVP |
| 7 | indexstore-db | 引用图/occurrence 强 | Phase 6+ 增强 |
| — | Periphery / UseGraph | 引用图、依赖图 | 借鉴思路 |

> 核心原则：确定性字段用官方工具；推断字段只输出候选 + evidence + confidence。

## 11. 性能采集与 .trace 解析

决策：可插拔 PerformanceBackend 架构。MVP 优先复用以 XcodeTraceMCP core，自研 raw xcrun 只作为 fallback。

| 优先级 | Backend | 判断 | 结论 |
|---|---|---|---|
| 1 | XcodeTraceMCP / `@xctrace-analyzer/core` | TOC/XPath/HAR export、hitches/hangs、not_exportable、Time Profiler | **MVP 第一候选**。优先评估是否直接依赖 core |
| 2 | instrumentsmcp | 录制+分析+report 一体化，CPU/SwiftUI/memory/hitches/launch/leaks/network | **横评对照**。Agent 控制录制+scenario profiling |
| 3 | apple-instruments-mcp | Python/uvx 轻量方案 | TS 方案不稳时子进程 backend 备选 |
| 4 | instruments-analyzer | DuckDB/Parquet 深度分析强 | Phase 6+ 大 trace/SQL 深挖 |
| 5 | memorydetective | memory/hang/jank/MetricKit/`.memgraph` 诊断强 | Phase 6+ 诊断增强 |
| 6 | py-ios-device | 实时指标采集（CPU/Memory/FPS/Jank），不读 `.trace` | FPS-like side-channel，标 approximate |
| — | raw xcrun xctrace export | 最小可控，但维护成本高 | **fallback**。不应默认主路 |

底层采集策略：
```
主指标（确定可采）：launch time / test duration / crash / hitches / hangs
近似指标（标注）：memory peak (approximate) / FPS (FPS-like)
实验性（保留原始 .trace）：xctrace summary 深解析
```

明确不用：
```
[不用] 链接 Apple 私有框架（TraceUtility，已归档）
[不用] 直读 .trace 二进制内部（speedscope 逆向路线）
[不用] XCTraceRunner（2021 停更）、mactrace（syscall 追踪）
[不用] flashlight 作为 iOS 采集（iOS 非主力）
```

## 14. 明确排除项与原因

| 技术/工具                        | 排除原因                                                 |
| -------------------------------- | -------------------------------------------------------- |
| Effect-TS 事件溯源编排           | 本地单机过度设计，学习曲线陡                             |
| fork OpenCode core               | private/未发布 + 绑 Effect/SQLite/coding 语义，改造≈重写 |
| 自研 provider 路由               | 非必要，AI SDK 已够用                                    |
| LangChain/LangGraph              | 偏重，与 TUI/MCP 组合不如 AI SDK 直接                    |
| Apple 私有框架(TraceUtility)     | 已归档，跨 Xcode 版本必坏                                |
| .trace 二进制逆向                | 私有格式，跨版本极脆                                     |
| XCTraceRunner / mactrace         | 停更 / 功能不对口                                        |
| tree-sitter-objc                 | grammar 不稳，仅 fallback                                |
| Maestro/Detox 作本地真机核心     | 本地真机支持弱 / 偏 RN 模拟器                            |
| 云真机/中心化服务/登录（第一版） | 与 Local-first 冲突，列为可选扩展                        |

## 15. 版本与依赖管理策略

```
- Bun 版本锁定（bun.lockb），CI 与本地一致
- Xcode/CLT 版本在 doctor 检测并记录到 run metadata
- xctrace/xcresulttool 输出 schema 跨版本容错，backend 吸收差异
- Appium + xcuitest-driver + WDA 三者版本需匹配，锁定并在 doctor 校验
- 第三方参考代码只借鉴不整包依赖，降低供应链风险
- 敏感依赖（pymobiledevice3, GPL 类）以子进程调用，避免许可证污染主工程
```

## 16. 选型风险与替代方案

| 风险                        | 影响          | 替代/缓解                                         |
| --------------------------- | ------------- | ------------------------------------------------- |
| OpenTUI 不成熟              | TUI 迭代受阻  | 抽象渲染层，回退 Ink                              |
| Bun 子进程管理坑            | 长任务泄漏    | 严格生命周期管理，必要时 Node 回退                |
| WDA/签名首次跑通难          | 用户流失      | 强 doctor 引导 + 端到端真机 spike                 |
| xctrace schema 跨版本变动   | 性能解析中断  | 版本容错 + 主推 hitches/hangs + 保留原始 .trace   |
| 元素定位/断言不稳           | 探索不可靠    | 人在环路录制，无断言不判 passed                   |
| provider 绑定单一           | 迁移成本      | provider adapter 抽象，OpenAI-compatible 优先     |
| 对话类 App（元素定位难） | 定位/断言更难 | 元素定位 spike，必要时退纯录制 + 流式完成断言原语 |
