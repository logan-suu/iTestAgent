# iTestAgent 技术选型文档

## 1. 目的与范围

本文件给出 iTestAgent 各层技术选型的**决策、候选对比、选择理由、风险与替代方案**，用于研发落地和评审。

- 产品形态：Local-first、TUI-first、只面向 iPhone 真机的测试 Agent
- 目标用户：iOS 客户端开发者本地自测与失败复现
- 参考验证项目示例：无既有测试、需自动构建签名的 iOS 项目
- 选型基准日期：2026-07；跨 Xcode 版本兼容以 Xcode 26 为参照

## 2. 选型原则

```
1. 复用优先：底层能力尽量复用成熟 repo，自研聚焦编排与语义层
2. 官方稳定优先：优先 Apple 官方 CLI（xcodebuild/devicectl/xctrace/xcresulttool）
3. 主流轻量优先：Agent/LLM/工具协议用主流标准（AI SDK + MCP），不自研底层
4. 本地优先：单机可用，不依赖服务器/云/登录
5. 可维护优先：不碰私有框架、私有二进制格式、停更工具
6. 诚实降级：研究级能力（探索/断言/FPS/xctrace summary）以尽力而为 + 明确降级
```

选型标记：

```
[采用]   直接依赖/集成
[借鉴]   参考其实现，不整包依赖
[自研]   无成熟替代，必须自建
[不用]   明确排除，附原因
```

## 4. 语言与运行时

决策：TypeScript + Bun [采用]

候选对比：

| 方案              | 优点                                                         | 缺点                                 | 结论                                       |
| ----------------- | ------------------------------------------------------------ | ------------------------------------ | ------------------------------------------ |
| TypeScript + Bun  | 启动快、原生 TS、内置 bundler/test/包管理、易分发单文件 binary、与 OpenCode 生态一致 | 部分原生模块兼容性需验证             | 采用                                       |
| TypeScript + Node | 生态最广、企业兼容好                                         | 启动慢、需额外 transpile/loader      | 备选/兼容回退                              |
| Swift             | 与 iOS 工具链最近                                            | Agent/TUI/LLM/MCP 生态薄，跨平台弱   | 仅用于必要的原生子进程                     |
| Python            | 脚本方便、部分解析库多                                       | CLI Agent/TUI/流式交互体验弱、分发重 | 仅个别 adapter（如 pymobiledevice3）子进程 |

理由：iTestAgent 是 CLI/TUI Agent，重交互、重工具调用、需易分发，Bun/TS 最契合，且能直接复用 AI SDK + MCP + OpenTUI 生态。

风险与缓解：Bun 长时间子进程管理（xcodebuild/appium/xctrace）需认真做信号转发与回收；必要时保留 Node 兼容运行路径。

## 5. 交互层：CLI 与 TUI

决策：CLI 用 yargs/commander（轻量入口）[采用]；TUI 用 OpenTUI + Solid [采用]

| 方案              | 定位                               | 结论                  |
| ----------------- | ---------------------------------- | --------------------- |
| OpenTUI + Solid   | 与 OpenCode 一致、组件化、全屏交互 | 采用（核心界面）      |
| Ink（React TUI）  | 成熟、生态大                       | 回退方案              |
| blessed           | 老牌、低层                         | 不用（老化）          |
| yargs / commander | 命令解析成熟                       | 采用（轻量 CLI 入口） |

理由：产品是 TUI-first，OpenTUI 与目标生态一致、可借鉴 OpenCode 的渲染/键位/插件槽；CLI 只做 doctor/devices/config/version 等轻量入口。

风险与缓解：OpenTUI 较新、文档薄，预留一次 TUI 框架回退（Ink）的抽象层。

## 6. 编排与 LLM

决策：自建 Agent 编排循环 [自研] + Vercel AI SDK [采用] + OpenAI-compatible provider [采用] + MCP TS SDK [采用]

关键结论：

```
- OpenCode 的编排内核是 private、未发布、绑定 Effect + SQLite 事件溯源 + coding 语义，无法 import、不建议 fork
- iTestAgent 照搬其循环结构与权限模型，用 AI SDK 多步 tool-calling 自建循环
- iPhone 真机能力以 MCP tools 暴露给循环
- 第一版只强支持 OpenAI-compatible API，provider 通过 adapter 可扩展 Claude/Gemini/本地模型
```

候选对比（编排实现方式）：

| 方案                                | 优点                                        | 缺点                                          | 结论         |
| ----------------------------------- | ------------------------------------------- | --------------------------------------------- | ------------ |
| AI SDK streamText 多步 tool loop    | 主流、轻量、onStepFinish/toolCalls 天然支持 | 高层封装，极端定制需下沉                      | 采用         |
| 自研 provider 路由（OpenCode 现状） | 完全可控                                    | 维护成本高，非必要                            | 不用（过度） |
| LangChain/LangGraph 等编排框架      | 功能多                                      | 偏重、抽象多、与 TUI/MCP 组合不如 AI SDK 直接 | 不用         |

理由：AI SDK + MCP 是最主流、最轻的组合，能以约 1/10 复杂度获得约 90% 收益，且工具解耦、易扩展。

## 7. 工具协议：MCP

决策：MCP TypeScript SDK [采用]

```
- iPhone 真机操作（devicectl/WDA/Appium/截图/日志/xctrace）封装为 MCP tools
- 工具抽象 { description, inputSchema, outputSchema, execute } + 输出大小限制
- 支持本地 stdio server；engine 用统一 tool 接口消费
- adapters 以 MCP 形式插拔，engine 不直接拼命令
```

理由：MCP 是工具生态标准，AI SDK 原生消费，解耦 engine 与底层工具，便于后续接入 Appium MCP/XcodeBuildMCP 等参考实现的工具。

## 8. 存储与配置

决策：SQLite + Drizzle [采用]（metadata）+ 文件系统（artifacts）+ JSONC 配置 [采用]

| 方案                     | 用途                                             | 结论                     |
| ------------------------ | ------------------------------------------------ | ------------------------ |
| SQLite + Drizzle         | session/run/step/flow/baseline/permission 元数据 | 采用                     |
| 文件系统                 | 截图/视频/日志/xcresult/trace 大文件             | 采用                     |
| JSONC + jsonc-parser     | 全局/项目分层配置，支持注释与 $schema            | 采用                     |
| 事件溯源(Event Sourcing) | OpenCode 用法                                    | 不用（本地单机过度设计） |

理由：本地单机场景 SQLite 足够；大文件走文件系统，DB 存索引；配置用 JSONC 支持分层与注释。

存储布局：

```
~/.itestagent/{config,db,projects,sessions,flows,baselines,runs}
runs/<run_id>/{plan.yaml,summary.md,result.json,artifact-index.json,artifacts/}
```

## 9. 真机执行技术栈

决策：Appium + appium-xcuitest-driver + WebDriverAgent [采用]；xcodebuild + xcrun devicectl [采用]

| 能力          | 选型                           | 结论               | 说明                             |
| ------------- | ------------------------------ | ------------------ | -------------------------------- |
| UI 自动化框架 | Appium Server                  | 采用（外部运行时） | 跨语言、WebDriver 协议           |
| iOS Driver    | appium-xcuitest-driver         | 采用（核心）       | 真机 XCUITest/WDA                |
| 真机桥        | WebDriverAgent                 | 采用（不自研）     | 设备端 WebDriver server          |
| 构建/测试     | xcodebuild                     | 采用               | build/test/xcresult              |
| 设备管理      | xcrun devicectl                | 采用               | list/install/launch（Xcode 15+） |
| 有测试项目    | XCUITest 路径                  | 采用               | 优先跑既有测试                   |
| 无测试项目    | Appium/WDA Agent Flow          | 采用               | 作为核心探索路径  |
| 设备底层      | pymobiledevice3                | 采用/可选          | syslog/crashlog/端口转发，子进程 |
| 备选设备工具  | facebook/idb、libimobiledevice | 借鉴/备选          | 依赖重或版本滞后                 |

排除：

```
[不用] idb 作为第一优先（依赖 companion，部署重，作备选）
[不用] Maestro 作为本地真机核心（本地真机支持弱，仅借鉴 flow 语义）
[不用] Detox 作为通用底座（偏 RN 模拟器）
```

签名/构建与日志：

```
[采用/外部] fastlane：cert/sigh/match/gym/scan，用于签名与构建 lane
[采用/外部] xcbeautify：xcodebuild 日志美化
```

真机前置约束（选型必须配套 doctor 引导）：

```
签名/Provisioning、Developer Mode、设备信任、WDA 首次安装
个人证书 7 天过期、bundle id 冲突、WDA 单独签名等坑
```

## 10. 项目分析技术栈

决策：结构识别用官方/成熟解析，语义用 SourceKit，链路只做候选

| 能力                 | 选型                                                  | 结论                              |
| -------------------- | ----------------------------------------------------- | --------------------------------- |
| Xcode 工程解析       | Tuist/XcodeProj + xcodebuild -list/-showBuildSettings | 采用                              |
| Swift 语法结构       | swift-syntax                                          | 采用                              |
| Swift 语义/符号/索引 | sourcekit-lsp / SourceKitten                          | 采用                              |
| 引用图/IB 资源       | Periphery                                             | 借鉴（规则参考）                  |
| ObjC 结构            | tree-sitter-objc                                      | 不用（grammar 不稳，仅 fallback） |
| 核心链路推断         | LLM + 证据                                            | 自研（只给候选，用户确认）        |

理由：工程结构可确定性解析，Swift 语义靠官方 SourceKit；核心链路无法从代码静态确定，改为“候选 + 证据 + 用户确认”。

## 11. 性能采集与 .trace 解析

决策：底层用 xcrun xctrace export [采用]；解析自研薄层 [自研]；成熟封装 [借鉴]

```
底层（必采）：xcrun xctrace export --toc / --xpath
  先 --toc 探测可导出 schema，再 --xpath 抽 hitches / hangs
  对 schema 名称/列做跨 Xcode 版本容错（尤其 Xcode 26 Deferred）
主指标：launch time / memory peak(近似) / crash / test duration / hitches / hangs
FPS：近似（FPS-like），从 Core Animation FPS / frame lifetimes 或 hitches 反推
xctrace summary 深解析：实验性，第一版保留原始 .trace 供人工打开
其它：XCTest metrics（launch/duration）、xcresultparser/xcparse（结果/附件）
```

可参考封装（读代码，不整包引入）：

| Repo                                   | 参考点                                        | 结论                   |
| -------------------------------------- | --------------------------------------------- | ---------------------- |
| jamesrochabrun/XcodeTraceMCP           | @xctrace-analyzer/core、hang 时间窗、诚实降级 | 借鉴（TS 可依赖 core） |
| nemanjavlahovic/instruments-mcp-server | parsers、xctrace 15–26 兼容、Xcode26 Deferred | 借鉴                   |
| jlreyes/instruments-analyzer           | SCHEMAS.md、hitches 方法论、DuckDB            | 借鉴                   |
| kyoya1123/instruments-profiler         | 真机流程、hitches 模板选择                    | 轻借鉴                 |

明确不用：

```
[不用] 链接 Apple 私有框架（TraceUtility，已归档，跨版本必坏）
[不用] 直读 .trace 二进制内部（speedscope 逆向路线，跨版本极脆）
[不用] XCTraceRunner（2021 停更）
[不用] mactrace（syscall 追踪，不是性能解析）
[不用] flashlight 作为 iOS 采集（iOS 非主力，仅参考评分/报告）
```

## 14. 明确排除项与原因

| 技术/工具                        | 排除原因                                                 |
| -------------------------------- | -------------------------------------------------------- |
| Effect-TS 事件溯源编排           | 本地单机过度设计，学习曲线陡                             |
| fork OpenCode core               | private/未发布 + 绑 Effect/SQLite/coding 语义，改造≈重写 |
| 自研 provider 路由               | 非必要，AI SDK 已够用                                    |
| LangChain/LangGraph              | 偏重，与 TUI/MCP 组合不如 AI SDK 直接                    |
| Apple 私有框架(TraceUtility)     | 已归档，跨 Xcode 版本必坏                                |
| .trace 二进制逆向                | 私有格式，跨版本极脆                                     |
| XCTraceRunner / mactrace         | 停更 / 功能不对口                                        |
| tree-sitter-objc                 | grammar 不稳，仅 fallback                                |
| Maestro/Detox 作本地真机核心     | 本地真机支持弱 / 偏 RN 模拟器                            |
| 云真机/中心化服务/登录（第一版） | 与 Local-first 冲突，列为可选扩展                        |

## 15. 版本与依赖管理策略

```
- Bun 版本锁定（bun.lockb），CI 与本地一致
- Xcode/CLT 版本在 doctor 检测并记录到 run metadata
- xctrace/xcresulttool 输出 schema 跨版本容错，adapters 吸收差异
- Appium + xcuitest-driver + WDA 三者版本需匹配，锁定并在 doctor 校验
- 第三方参考代码只借鉴不整包依赖，降低供应链风险
- 敏感依赖（pymobiledevice3, GPL 类）以子进程调用，避免许可证污染主工程
```

## 16. 选型风险与替代方案
